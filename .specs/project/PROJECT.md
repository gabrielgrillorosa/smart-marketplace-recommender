# Smart Marketplace Recommender

**Repository:** `https://github.com/gabrielgrillorosa/smart-marketplace-recommender`

**Vision:** A hybrid AI-powered product recommendation system for a B2B multi-tenant marketplace, combining semantic RAG over product catalogs with neural collaborative filtering — inspired by the BEES (Ambev) marketplace architecture that serves suppliers like Nestlé and others across Brazil, Mexico, Colombia, Netherlands, and Romania.

**For:** Software engineers and technical recruiters evaluating full-stack AI engineering competence at Staff level; secondarily, any B2B marketplace that needs intelligent product recommendation at scale.

**Solves:** Cold-start and relevance problems in B2B marketplace recommendation — where traditional collaborative filtering fails because purchase history is sparse and product descriptions matter as much as purchase patterns. Combines semantic similarity (what the product *is*) with behavioral signals (what similar clients *bought*).

---

## Goals

- **G1 — Publishable portfolio project:** Deliver a fully functional, documented, Docker-composable project that runs end-to-end with `docker compose up` and demonstrates senior/staff-level AI engineering competence for a public GitHub repository.
- **G2 — Validate hybrid recommendation quality:** Demonstrate measurable improvement in recommendation relevance by combining HuggingFace semantic embeddings with a neural model trained on purchase history, compared to either approach alone.
- **G3 — Demonstrate polyglot architecture:** Show clean separation between a high-throughput Java/Spring Boot API (domain + scale) and a TypeScript/Node.js AI service (embeddings + RAG + model inference), communicating over HTTP — mirroring patterns used in real enterprise AI systems.
- **G4 — Zero cost to run:** The entire system runs locally with free-tier tools only (HuggingFace local embeddings, OpenRouter free LLM, Neo4j Community, PostgreSQL OSS), removing any barrier for evaluators to clone and run.

## Tech Stack

**API Service (domain + scale layer):**

- Language: Java 21
- Framework: Spring Boot 3.3
- Database: PostgreSQL 16
- Key libs: Spring Data JPA, Spring Web, Springdoc OpenAPI 3, Spring Actuator, Micrometer, Caffeine cache

**AI Service (intelligence layer):**

- Language: TypeScript (Node.js 22)
- Framework: Fastify 4
- Key libs: `@langchain/community` (Neo4j vector store, LangChain pipeline), `@langchain/openai` (OpenRouter LLM), `@xenova/transformers` (HuggingFace local embeddings), `@tensorflow/tfjs-node` (neural model training + inference)

**Graph + Vector Store:**

- Neo4j 5.x Community Edition
- Nodes: `Product`, `Client`, `Category`, `Supplier`, `Country`
- Edges: `BOUGHT`, `BELONGS_TO`, `SUPPLIED_BY`, `AVAILABLE_IN`
- Vector index: `product_embeddings` (384 dims, cosine similarity, MiniLM-L6-v2)

**Relational Store:**

- PostgreSQL 16
- Tables: `products`, `clients`, `orders`, `order_items`, `suppliers`

**LLM:**

- Provider: OpenRouter (`https://openrouter.ai/api/v1`)
- Model: `mistralai/mistral-7b-instruct:free` (zero cost)
- Use: RAG answer generation over product catalog in pt-BR and en

**Frontend:**

- Framework: Next.js 14 (App Router)
- Styling: Tailwind CSS 3
- Use: Functional demo UI — product catalog, client profile, recommendations, RAG chat

**Infrastructure:**

- Docker Compose (single command: `docker compose up`)
- Services: `postgres`, `neo4j`, `api-service`, `ai-service`, `frontend`

## Scope

**v1 includes:**

- Synthetic dataset seed: 50+ products across 5 categories (beverages, food, personal care, cleaning, snacks), 3 suppliers, 5 countries, 20+ clients with realistic purchase histories
- Product catalog CRUD API (Spring Boot) with pagination, filtering by category/country/supplier
- Client profile API (Spring Boot) with purchase history
- Embedding generation: HuggingFace `sentence-transformers/all-MiniLM-L6-v2` for all products, stored as Neo4j vector properties
- Neo4j graph model: product relationships by category, supplier, and country
- RAG endpoint: natural language queries over product catalog (`POST /rag/query`)
- Neural recommendation model: dense network (embedding → 256 → 128 → 64 → 1 sigmoid) trained on client purchase history, using HF embeddings as input features (not one-hot encodings as in parte05)
- Hybrid recommendation endpoint: top-N products combining semantic similarity score + neural model score (`POST /recommend`)
- Spring Boot proxy endpoint: `GET /recommend/{clientId}` delegates to AI service and returns ranked product list
- Next.js frontend: catalog view, client selector, recommendation panel, RAG chat panel
- Full OpenAPI/Swagger docs auto-generated by Springdoc
- Spring Actuator + Micrometer metrics on `/actuator/metrics`
- `docker compose up` brings all services with health checks
- README with architecture diagram, tech decisions rationale, and 5-command quickstart

**Explicitly out of scope:**

- Authentication / authorization (no login, no JWT — not the focus)
- Real BEES ONE data or any Ambev proprietary data
- Replicating the BEES ONE visual design or UI/UX
- Production deployment (Kubernetes, cloud infra)
- Real-time streaming recommendations (Kafka, WebSocket)
- A/B testing framework
- Model versioning / MLflow / experiment tracking
- Fine-tuning the HuggingFace model on domain data
- Multi-language LLM responses beyond pt-BR and en

## Constraints

- **Timeline:** Portfolio deliverable — scope is bounded to what one engineer can complete with quality in focused sessions.
- **Cost:** Zero — no paid APIs, no cloud services, no paid tiers. Everything runs locally on developer hardware.
- **Data:** Synthetic only. No web scraping, no real client data, no proprietary datasets. Open Food Facts may be used as optional enrichment for product descriptions only.
- **Reproducibility:** Any engineer cloning the repo must be able to run the full system with `docker compose up` and see a working demo in under 10 minutes.
- **Academic context:** Project is the capstone deliverable for the post-graduation course *Engenharia de Software com IA Aplicada* (modulo01), under the instruction of Erick Wendel (Google Developer Expert, Node.js core contributor). The TypeScript choice for the AI service reflects the course's TypeScript-first AI stack and the student's existing expertise — a deliberate, documented architectural decision.
